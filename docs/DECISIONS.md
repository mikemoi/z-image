# 关键工程决策

## kind 与 entry_type 分开

- `kind`：从哪个文字入口进入，当前为 `idea/log/plan`。
- `entry_type`：内容本身是什么，固定为想法/句子/规则/决策/知识/资料/记录。

因此 `kind=idea, entry_type=句子` 是合法且常见的组合。

## theme 与 domain 暂时并存

`theme` 是旧截图体系，可生长并被 Browse、Vision、精选标签链路使用；`domain` 是统一五维里的固定大领域。当前保留两者是为了不破坏已有主流程。未来迁移必须先处理 Browse、stats、promote、Vision prompt 和历史数据，不能直接删列。

## 来源不表示可信度

`source` 只表示进入系统的路径：自己/截图/文件。“他人经验”属于 `topics`，不是来源。文字 Entry 有 `source` 列；截图天然来源为截图，目前不重复存列。

## 人工分类优先

人工修改分类字段时，API 将 `ai_classify_status` 置为 `done`，后台分类器不再覆盖。重新分类会清空分类并改回 `pending`。

## 删除语义

- 所有普通“删除”只移入回收站，不弹永久删除确认。
- “我的 → 回收站”统一恢复和永久删除；永久删除必须二次确认。
- 截图永久删除时，只有原图没有其他 item 引用才删除磁盘文件。

## 人工重点优先

- `highlights` 保存逐字原句，不保存位置索引，避免正文轻微修改后错位。
- AI 自动分类顺便建议最多 3 条重点，不增加独立调用；已有人工重点时 Worker 不覆盖。
- 用户先保存内容，再通过独立“标重点”入口增删重点；重点标注不混入首次输入或分类编辑。
- 手机端在用户选中文字时立即记住选区，点击“标为重点”时不再依赖仍然可见的系统选区。

## AI 先于人工编辑

- 新内容保存后由后台自动分类、打标签并建议重点，正常流程不要求用户主动点 AI。
- 人工编辑器只修改正文与分类，不再放“让 AI 重新分类”；兼容 API 暂时保留。
- 截图“问问 AI”是独立理解入口，放在人工标签、重点和编辑之前。

## 批阅以阅读为主

- 今日推荐与集中批阅直接完整展开 OCR 原文，不使用内部滚动框。
- “问问 AI”紧跟在识别原文下方；批阅页取消编辑跳转和精选，只保留重新分类、删除、前后翻页与写想法。
- 截图精选后端继续兼容，但前端不再暴露入口；全部已保存内容本身就是可搜索资产。

## 配置优先级

模型配置优先级为 `core.settings` > 环境变量默认值。设置页保存后立即生效，无需重启。三个模型共用同一进程内每日预算计数。

## 时间内容分为日志、近日和长期计划

三者不是同一种“计划”的长短期层级，而是不同生命周期：

- 日志：已经发生的事。
- 近日：近几天需要落实或记住的轻量备忘，也可作为购物清单。
- 长期计划：长期方向、人生阶段和长期规划。

信息架构暂定：

```text
记录
├─ 日志
└─ 近日

我的
├─ 长期计划
├─ 数据概览
└─ AI 设置
```

“近日”产品名固定，内部入口类型建议使用 `kind=memo`，不要复用长期计划的 `kind=plan`。

近日的交互边界：单行快速添加；点击左侧圆圈后文字加删除线、变淡并移到下方；可再次恢复；完成项不立即消失；支持手动清除已完成。第一版不做截止日期、优先级、提醒、子任务、进度、逾期红字或剩余数量，避免演变成任务管理器。

## Entry 卡片普通状态只读

想法、日志和长期计划采用同一交互：正文与 topics 在上，底部一栏显示“类型 · 领域 · 用途 · 来源”，右侧只有编辑和删除。普通卡片不提供分类下拉、标签删除、添加标签或重新分类，避免误触。

点击“编辑”后一次修改正文、类型、领域、用途和标签；日志额外允许修改日期和用户填写的心情。来源由系统确定，只展示不修改。编辑器内保留“让 AI 重新分类”：先保存最新正文，再清空分类并置 pending。普通保存代表人工确认，AI 不再覆盖。

想法默认就是一等内容，不需要再点“精选”。前端移除精选按钮，但后端 promote 端点与旧字段暂时保留兼容；想法不会因此自动复制进 `core.knowledge`。

想法与日志都显示时间，但含义不同：想法使用不可编辑的 `created_at` 表示灵感或感悟出现的时间；日志使用 `logged_for + created_at` 组织一天里发生的各种记录。编辑想法不改变其原始创建时间。

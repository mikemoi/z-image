# v0.3 规划 · 从"能存"到"会整理" + 新入口(文字)

> 本文档整理自与用户的长期讨论,记录下一阶段要做的事、为什么这么做、以及已经拍板的红线。
> 现状:z-image 五步已完成(骨架/上传/Vision/前端/消化闭环),部署就绪,用户已在轻度日常使用、大量直接上传不做人工整理。

---

## 0. 全局心法(贯穿本文档所有功能)

- **拉,不是推**:AI 建议只在用户主动打开/点击时出现,绝不做带计数的"建议中心"、绝不做上传后自动跑一堆建议。看完不理它就没了,不累积、不计数、无愧疚。
- **AI 建议、用户拍板**:AI 干脏活(判断/建议/归类),决定权永远在用户,一键采纳或忽略。
- **显示层折叠,绝不物理删除**:任何"合并/去重/拼接"都只是浏览时收起来,底层数据(原图、原记录)照留。原图是唯一副本,自动删除近似项是不可接受的风险。
- **原文事实源 vs AI 补充,必须分开标注**:AI 生成的解释/看法可能出错,不能和用户存的原文混为一谈。
- **手动修正优先于 AI 重跑**:任何批量重新处理/重打标签,绝不覆盖用户手动改过的字段。
- **记录是为了看见规律,不是打卡自责**:任何日志/习惯/用药记录功能,禁止 streak、禁止"你已 N 天没做"式提醒。这条对 ADHD 用户尤其重要。
- **原图三份备份已确认稳定**:磁盘 + 磁盘备份 + Google 相册(独立异地)。数据库(标签/分类/知识)是否单独备份,建议但非阻塞项。

## 2. v0.3「整理版」—— 让库越用越干净

### 2.1 底座:向量(pgvector)
- 每条内容算一次 embedding,同库同备份,成本低(不同于"AI 看法"那种每次调用都烧钱)。
- 架构早年已预留 `embedding vector(1536)` 注释,直接启用即可,不引入 Qdrant(对单人自托管是纯负债)。
- **必须和"存量导入"(5-6000 张历史图)同一版做**:光灌存量不做语义聚合,等于造一个更大的乱库;向量聚合正是让存量可用的前提。

### 2.2 统一交互入口:详情页「问问 AI」
所有智能整理功能,都通过这一个按钮触发(用户点击才调用,按需付费):

1. **看法 / 定义**(优先做,已拍板的第一味)—— ✅ **第一批已实现**
   - 解释看不懂的术语/内容,给"这张图在讲什么 + AI 的看法"。
   - 明确标注为「AI 补充」,与原文/原图分开展示,不混淆事实源。
   - **附带质量判断 + 标签**(用户后加的需求,详见 2.2.1)。
   - **附带分类建议 + 一键采纳**(生长的分类第一步,详见 2.3):现有分类装不下时提议新 theme,用户点「采用」即建 tag + 归入。
   - **成本控制**:详情页主动点「问问 AI」才调用(拉不推);结果缓存进 `image.items.ai_insight`,再看不重复烧钱;`refresh=true` 才强制重算。不受每日 Vision 预算影响(低频、用户主动)。
   - 暂不做「推荐操作」(建议入脑/删除/归类的一键采纳):用户目前是轻度使用、连入脑都懒得做,待办式建议只会添堵。等用户真正开始觉得"整理烦"了、常态化使用后再加。
   - **实现落点**:后端 `vision.call_insight` + `POST /api/items/{id}/insight`(缓存)+ `POST /api/items/{id}/adopt-theme`;前端 `Detail.jsx` 的「问问 AI」区块;新列 `image.items.ai_insight`(`db.ensure_schema` 运行时迁移 + `init.sql`)。
   - **已端到端验证**(2026-07-04,临时本地库 + 真实 AI):造一张健身截图 → Vision 归类 → 「问问 AI」返回看法(quality=干货)+ 提议新分类「运动」→ 一键采纳后 tag 生长、条目改归「运动」;搜索「健身/大肌群/深蹲」均命中(旧版会全空)。验证用的临时库/图/venv 已清理。

2. **关联**(向量近邻,先做,最便宜)
   - 打开一条,底部带出"相关的还有这几条"(纯最近邻,不需要聚类算法)。
   - 知识类文章:只关联,不折叠——两篇讲复利的文章是互补,不是重复,别把一篇藏起来。

3. **合并显示**(聚类成组,后做,更难)
   - 相似内容折叠成一张卡:"合并了 N 条",点开可展开,数据全在。
   - **碎片/金句要狠狠折叠**:同一句"放低期待"存了 10 次 → 显示 1 张"合并了 10 条"(去噪主战场)。
   - 对应架构里预留的 `core.topics`(主题容器,从高频 topic 标签自然长出)。

4. **长文拼接**(同类问题,同一套机制)
   - 现状:长文被拆成多张连续截图时,各自独立存,不强行拼接(残缺被承认)。
   - 判断信号:同一批上传 + 版式/语义高度相似 + 文字能首尾相接。
   - 三个信号叠加即可判断,显示层拼成一篇完整文章,原图照留(不是删除合并)。

#### 2.2.1 AI 看法的质量判断(标签)
- 用户提出:AI 给看法时,顺手判断"这条是干货 / 鸡汤 / 无信息量",帮助用户挑出噪音、激活"提纯"这根支柱。
- **关键红线,不能一刀切**:"鸡汤"≠"没用"。架构里早有明确判断——情绪帖("十年没头绪""3次爆仓")对用户是**反面教材/群体样本**,有提炼价值,该归 `避坑`,该入脑;只有纯无信息量的("写得真好"评论流)才是真垃圾(`useless`)。
- 所以标签要分层:
  - 纯无信息量 → 真垃圾,建议可删(接上现有 `status='useless'`)
  - 有反面/心态价值的"鸡汤" → 其实是 避坑/心态,该留
- **倾向的实现方式**:存下来 + 可按标签筛选(比如"给我看所有 AI 判为无信息量的"),支持用户主动发起的"清库仪式"——而不是做成被动的待清理计数(避免变成新的焦虑源,与全局心法冲突)。

### 2.3 生长的分类(替换固定 theme 列表)
- **根因**:任何预设分类对个人第二脑都是错的——运动、情绪、健康、旅行……写死的列表永远跟不上真实生活。
- **方案**:
  - `theme`(主题)→ 开放生长:AI 遇到不认识的领域时建议新建分类,用户点头才生效;首页网格显示"用户的高频分类"(按实际出现频率排序),而非硬编码列表。
  - `use`(用途)→ 保持稳定不变:方法/避坑/心态/工具/灵感(小而够用,"情绪"可归入心态)。
- **护栏**:AI 优先从已有分类里挑,确实全新才提议;用户确认或合并相似分类;低频一次性内容不进首页网格,避免网格膨胀。
- **与向量聚类是同一件事**:向量聚类可以主动发现"你有一簇内容都在讲运动",提议建新分类,用户确认即诞生——这样存量导入时,分类是从真实内容里长出来的,而不是硬塞进错格子。

#### 2.3.1 无向量版「自动判候选 + 聚合冒泡」—— ✅ **已提前实现**(2026-07-04)
- **动机**:用户明确"我可能懒得点『问问 AI』"→ 分类生长不能只靠主动拉,否则形同虚设。改为挂在**每张上传后本来就在跑的那次自动 Vision** 上,零额外调用。
- **实现**:
  1. `vision.py` PROMPT 加第 8 项 `suggested_theme`:`life/other` 视为"兜底大类",内容能归到更具体领域(运动/情绪/健康…)时即使勉强能塞进兜底也给出具体候选;`normalize` 提取该字段,worker 原样存进 `ai_output`。
  2. `GET /api/stats/theme-candidates?min=3`:聚合 `ai_output->>'suggested_theme'`、排除已采纳(theme=候选)、达阈值才返回。**不逐条推,攒够一簇才冒一次**。
  3. `POST /api/stats/theme-candidates/adopt`:建 tag + 把整簇批量归入。
  4. 前端 `Browse.jsx` 顶部冒泡卡(建/忽略),忽略走 `localStorage` 记住、不再冒(偶遇不催办)。
- **已验证**:真 AI 跑健身图 → `theme=life` + `suggested_theme=健身`(修 prompt 前太保守返回 null,修后正常);聚合(运动=5 冒出、情绪=2 被阈值挡)、批量采纳(归入 5 条 + 建 tag)、采纳后不再冒泡,断言全过。
- **已知局限(留给向量批)**:模型对同类可能给不同措辞(健身/运动/锻炼)→ 文本聚合会分散;**语义归一靠 2.1 的向量聚类解决**,现文本版作起点够用。

### 2.4 批量整理(配合上面几项)
- 触发场景:分类升级后重新归类、大规模重打标签、存量导入后的整理。
- **铁律**:绝不覆盖用户手动修正过的标签(人工修正优先级最高)。
- **稳妥流程**:操作前拍数据库快照 → 先预览"会改哪些",用户确认后再应用 → 有条件时先在副本/测试环境跑一遍。

### 2.5 存量导入(5-6000 张历史图)
- 必须和向量聚合、生长分类同一版做,否则灌存量 = 造一个更大的乱库。
- 沿用架构里早定的原则:限速慢跑、按日预算、不淹没在待 review 里。

## 3. 新入口:文字(第二个真正的入口,复用 core)

### 3.1 定位决策:不新开平行表,全部并入 `core`
- 原因:
  1. `core` 本来就是为跨入口设计的("脑本体"),`core.notes.source_id` 早就允许为空(手敲的没有来源)。
  2. 用户手写的想法/经验是**即成品**,不需要 `image.*` 那种三层加工结构(文件→提取→清洗)——照猫画虎建镜像表纯属浪费。
  3. v0.3 正在给 `core` 建向量/搜索/聚类基建——放进 `core` = 自动获得搜索、重新遇见、与截图跨来源关联(比如自己写的止损心得和存过的止损截图自动凑到一起,这是第二脑的命脉);单独建表则要重新实现一遍,且永远不会和截图关联。
- **区分方式**:不靠新表,靠来源标记——扩展 `core.sources` 的 origin 类型(`image` / `manual` / `ai-clip` / `experience` ……),随时可按来源筛选,但共享同一套搜索/聚类。
- **落点**:随手的想法 → `core.notes`(轻,靠"重新遇见"轮换);深思的经验/知识/AI 精华 → `core.knowledge`(可切块、可搜索、可入脑)。

### 3.2 速记 Inbox(轻量捕捉层,已拍板要做)
- 用户已证明是"先扔、以后再整理"的人类型 → 记想法也必须零摩擦,不能当场逼决定"这是知识还是碎片、打什么标签"。
- 但生想法不能直接倒进 `core`(会稀释精选脑),需要一个和 `image.items` 对称的"加工台":
  ```
  z-image 入口:  截图 → image.items(加工台)→ 消化 → core
  速记入口(新): 想法 → 捕捉表(加工台)  → 消化 → core
                 （就 text + status + 来源类型,
                  没有文件、没有 OCR,超轻）
  ```
- 捕捉:一个"记一条"的框,写完就走,状态"待整理"。可能接 iOS 快捷指令/分享扩展。
- 消化:用户或 AI 有空再归位——深思的 → `core.knowledge`,随手的 → `core.notes`。走和截图一样的消化路径(review/promote)。

### 3.3 长远重要内容:五年/十年计划
- 属于"脑"(知识)层,但要**钉住/加重要标记**,不能沉到库底靠搜索才找得到——它是北极星,应主动摆在眼前(比如首页固定位置)。

### 3.4 四层框架(用于以后归类新想法,避免混淆)

| 层 | 是什么 | 例子 | 靠什么找 |
|---|---|---|---|
| 🧠 脑(知识) | 你知道/学到/想留的 | 截图、想法、经验、五年十年计划 | 按意义(搜索/关联) |
| 📔 日志(事件) | 某天发生了什么 | "今天丢人了"、情绪、日记 | 按时间(翻日期) |
| 📊 记录(数据) | 结构化、能统计 | 吃药时间、吃了啥、作息 | 时间轴 + 统计 |
| 🛠 工具 | 跑在上面三层之上 | 自动 CBT | 读数据、也写回 |

四层共享同一条时间轴 + 同一套标签,才能做"组合分析"(如"漏药那几天日志是不是更丧")——这是不各用各 App、放进一个 zbrain 的最大价值。

### 3.5 日志(下一个要落地的新功能,已定方向为"近期做")
- **核心**:日志 = 带日期的文字,主轴是时间(区别于其它内容靠"意义"检索)。
- **写**:必须 3 秒搞定,一个框"今天怎么样?",自动盖今天日期,可接 iOS 快捷指令/锁屏。
- **真正价值在"回看",不在"写"**:时间拉开距离本身有疗愈力("现在看根本不算啥")。要让回看舒服:
  - 按日期翻的时间线
  - "一年前的今天你写了……" 温柔冒出来(和"重新遇见"同一气质:偶遇,不催办)
- **红线(已死守)**:绝不做 streak、绝不提醒"你已 N 天没写"。想写就写,不写就没,零负担。
- **通往 CBT 的桥(远期,v1 不碰)**:攒够日志后,AI 能帮忙看规律("这周你反复写'来不及'")——这是自动 CBT 的原料,但地基(写/翻/重现)先打好,CBT 以后单独设计,碰心理更要慢、要稳。
- **待用户拍板的开放问题**:要不要加一个可选的「心情」轻标记(emoji/一档好坏)?
  - 好处:是以后和"用药/作息做组合分析"的钩子("漏药那天心情如何")。
  - 前提:必须完全可选、一点即好、绝不做成"每天必须打卡心情"。

### 3.6 结构化记录 + CBT 工具(远期,单独设计,不在本版范围)
- 用药时间、吃了什么、作息 → 需要自己的结构化 schema,不能塞进文字笔记里。
- CBT 工具设计要格外谨慎(碰心理),等结构化记录跑起来、日志攒够素材后再单独规划。
- **红线预告**:任何用药/习惯记录,只用于"看见规律",绝不做打卡、绝不做愧疚提醒。

## 4. 优先级与顺序建议

1. **近期(用户已在攒真实内容,先用着)**:文字速记 Inbox + 日志——这两者形态几乎一致,可一起做,离现状最近。
2. **v0.3 整理版(一次做透)**:向量底座 → 关联(先做,便宜) → 生长分类 → 长文拼接 → 合并聚类(后做,较难) → 配合存量导入 + 批量整理。
3. **快修**:搜索覆盖 bug、缺失分类止血贴——用户一句话确认即可先做,不必等大版。
4. **远期(单独规划,不急)**:结构化记录(用药/作息)、自动 CBT 工具。
